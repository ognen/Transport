Transport

Swiss public transport API 2.0 (draft 3)

Introduction

The Transport API allows interested developers to build their own applications using public timetable data, whether they're on the web, the desktop or mobile devices. Find out more about how you can start exploring the timetable data playground or just browse the API Documentation.

The source code can be found on GitHub, please ask any technical questions there. If you need a direct contact write a mail to transport@opendata.ch.

General

Hyperlinking and Discovery

The version 2 (v2) API stipulates that all navigation from responses to new requests should be done by reading and interpreting the embedded links in the JSON objects.

E.g. if one knows the id of a station (e.g. 4556), one is not allowed to immediately construct the URL /stationboards?id=4556 but should follow the embedded stationboards link in the response obtained by querying a locations endpoint.

The encoding of links is explained in the link object section.

Metadata and Extensibility

Implementations of this specification are able to extend the specified objects with data that is unique to them, and at the same time they must not confuse client of the API with properties that mean one thing in one and something else in another implementation.

To do so, the following rules are to be followed:

  • We distinguish between global identifiers and namespaced identifiers:
    • a global identifier is a plain name (that doesn't contain the / character)
    • a namespaced identifier is in the form namespace/localName
  • All global identifiers for object properties and enumeration values are reserved for the specification, implementations must add only namespaced properties to the specified objects also extend the enumerations only with namespaced values.
  • Namespaces should allow for enough diversity and therefore preferably follow the "inverse domain notation" rule, e.g. com.example.
  • Client implementations should simply ignore properties they know nothing about.

API Resources

The Transport API builds on REST style resources which respond in JSON. Individual resources are detailed below.

Root Resource

The root resource is used as the only entry point in the API. Implementations publish just this URL. Its primary concern is to reveal the capabilities of the API provider.

This resource must be provided by every implementation.

Resource URL (of the reference implementation)

http://transport.opendata.ch/v2

Request Parameters

None.

Response Object

Property Name Description
version The version of the API that the implementation provides.
author Identifies who implements this API.
locations A link object pointing to the global locations resource. Required.
stationboards A link object pointing to the global stationboards resource. Optional
connections A link object pointing to the connections resource. Optional.

Example Request

GET http://transport.opendata.ch/v2

    }

In this example we can see that the server implements all the resources of the API.

The Relation of the link to its containing object

The meaining of a link is always determined in relation with its containing object. The relation between the containing object and the embedded link is given by the property name under which the link is contained within the parent.

For example, in the object:

{
    "name": "Zurich HB",
    "type": "station",
    "stationboards": {
        "href": "./stationboards/123456"
    }
}

We interpret that: the (relative) URI ./stationboards/123456 points to a stationboards resource for the station Zurich HB, because the link object is found under the property stationboards of the station object named "Zurich HB".

Similarly

{
    "name": "Zurich",
    "type": "locality",
    "locations": {
        "href": "./locations/zuerich"
    }
}

is interpreted as: the link ./locations/zuerich points to a locations resource where all the locations within the city of Zürich can be found, because the link object is found under a locations property inside a the Zurich locality object.

Self-Links

A self-link is an object (also a link) pointing to a resource of the same object, usually in greater details. It is usually used in collection resources (such as trip objects in stationboadrs to point to the full-version of the trips themselves.

For example:

{
    "stationboards": [ {
        "trip": {
            "href": "./trips/123456",
            "line": "2",
            "category": "bus"
        },
        "station": {
            "id": "8502996",
            "name": "Aarau, Bahnhof"
        }
    } ]
}

In this response, the trip object contains a self link (can also be said that it is a self link, because it is an object containin a href property) to a resource that represents the full trip.

The timing object can be used to represent both plan and realtime data.

Name Description Example
platform The plannned/estimated arrival/departure platform 8
departure The departure time to the checkpoint
Date format: ISO 8601		 2012-03-31T08:58:00+02:00
arrival The arrival time to the checkpoint
Date format: ISO 8601		 2012-03-31T09:35:00+02:00
capacity1st The occupation load of 1st class coaches 1
capacity2st The occupation load of 2nd class coaches 2

Stop Object

The stop object represents a stopping of one one trip on a station at a given point in time.

Name Description
station A location object showing this line's stop at the requested station.
trip A trip object denoting the trip (vehicle) that makes the stop.
plan a timing object representing the planned stopping times.
prognosis a timing object representing the prognosis over on when this stopping would take place. Optional.

Trip Object

The trip object represents the a single trip performed by a transportation vehicle over its usual route.

Name Description Example
id The id of the trip. 9876543
href An URL (self link) pointing to the complete trip resource. /v2/trips/9876543
category The type of the trip. One of these. bus
number The number of the connection's line, e.g. Bus line 2RB: line instead of number? 2
operator The operator of the connection's line BBA

You can limit the response data with the fields parameter. If any fields[] is passed, only the specified fields are in the response. The API is aware of the hierarchy, if you set a field, everything below is in the response unless a more specific (further down in the hierarchy) field is set.

Example Request

GET http://transport.opendata.ch/v1/connections?from=Lausanne&to=Genève&fields[]=connections/from/departure&fields[]=connections/to/arrival

Example response

{
    "connections": [ {
        "from": {
            "departure":"2012-05-01T19:42:00+0200",
        },
        "to": {
            "arrival":"2012-05-01T20:15:00+0200",
        }
    },
    {
        // other connections...
    } ]
}
